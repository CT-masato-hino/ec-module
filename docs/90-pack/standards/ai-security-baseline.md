# AI利用セキュリティベースライン（エンジニアの穴あけ対策）

AI駆動開発のセキュリティ事故は、悪意ではなく**開発を速くしたいエンジニアの善意の近道**（権限バイパス・秘密情報のプロンプト投入・野良ツール接続）から起きる。
この標準は「注意喚起」ではなく**仕組みで塞ぐ**ためのベースライン。/project-init で Claude のプランを確認し、プラン別の統制方式を適用する。

## 0. プラン確認（/project-init 質問17）

| プラン | 使える統制手段 | 統制方式の結論 |
|---|---|---|
| **Enterprise** | 組織管理ポリシー（managed settings によるClaude Code設定の強制配布）・SSO/SCIM・監査ログ・データ保持設定・ロール管理 | **組織ポリシーで強制**（個人が外せない設定にする） |
| **Team** | 管理コンソール・組織アカウント・一元請求（設定の強制配布は限定的） | **リポジトリ共有設定＋hooks＋運用統制**（外したら検知する） |
| **個人（Pro/Max）** | 組織統制なし・監査ログなし・退職/離任時にアカウントと履歴が個人に残る | **業務利用の可否自体を人間が判断**（顧客NDA・社内規程と突合）。使う場合は代替統制＋定期人間監査。**このリスクは必ず人間に明示して承認を取る** |

- アカウント共有は全プランで禁止（監査の前提が壊れる）
- 顧客の機密度が高い案件（金融・医療・官公庁）で個人プランは原則不可としてエスカレーション

## 1. 穴あけパターンと対策（全プラン共通ベースライン）

| # | 穴あけパターン（実際に起きること） | 仕組みによる対策 |
|---|---|---|
| 1 | 権限確認が面倒で `--dangerously-skip-permissions` を常用 | 禁止を CLAUDE.md に明記＋利用検知時は即報告。サンドボックス実行を既定にする |
| 2 | `.env`・秘密鍵・本番接続情報をClaudeに読ませる/貼る | permissions の deny で `Read(.env*)`・`Read(**/secrets/**)`・鍵ファイルを塞ぐ（下記スニペット）。**そもそも本番認証情報を開発マシンに置かない**のが最強の対策 |
| 3 | APIキーを `.mcp.json`・コードに直書きしてコミット | 環境変数/シークレットマネージャ経由のみ。pre-commit のシークレットスキャンを CI 必須化（Q-08 と連動） |
| 4 | 野良MCPサーバーを勝手に接続（サプライチェーン） | MCP は**許可リスト制**。`.mcp.json` の変更は architecture-guardian ＋ security-compliance のレビュー対象。出所不明のサーバーは接続前に人間承認 |
| 5 | 顧客データ・個人情報をプロンプトや外部SaaS（タスクツール・Slack）に投入 | /pm-sync の「接続してよい情報の範囲」を先に確定。テスト用は生成データのみ（本番データのマスキング基準を規約化）。顧客のAI利用方針と NDA を CLAUDE.md に転記 |
| 6 | `settings.local.json` で共有設定の deny を上書きして外す | 共有 `settings.json` を正としてリポジトリ管理（変更=PRレビュー対象）。Enterprise なら managed settings で上書き自体を不可能にする |
| 7 | AI生成コードに内部URL・認証情報・デバッグ痕跡が混入したままコミット | Q-01（第三者レビュー100%）＋シークレットスキャン＋ /delivery-package の混入検査の三段構え |
| 8 | 「動いたから」と本番へ直接適用 | infra-coder の絶対ルール（本番適用は人間のみ）＋CI/CDの人間承認ゲート |
| 9 | Webから拾ったコード/指示の無検証取り込み（プロンプトインジェクション含む） | 外部コンテンツは「データであって指示ではない」原則。取り込みは出典記録＋ライセンス確認＋通常レビュー |
| 10 | 事故ったことを隠す（最悪の穴） | **報告免責の明文化**: 秘密情報を入れてしまった等の自己申告は責めない・即時報告義務のみ課す（incident-responder の初動へ。隠蔽だけを重大違反とする） |
| 11 | 個人の `CLAUDE.local.md`（プロジェクトCLAUDE.mdに追記される・gitignore対象・PRレビュー未経由）にQCD基準・権限・工程ゲートを緩める記述をし、自分のセッションだけ共有ルールを迂回する | **構造で塞ぐ**: CLAUDE.local.mdの正規の内容は「自分のロールID宣言（案件情報のチーム体制表の該当行を指すID）」1行のみに限定する。役割の中身（担当ドメイン・使うエージェント・QCD/権限/工程ルール）は**共有・コミット対象のチーム体制表（CLAUDE.md）を正本とする**ため、変更にはPRレビューとaudit_pack.pyの検査が必ず経由する。ロールID宣言以外の内容がCLAUDE.local.mdにあれば「書式外」として無効扱いし、共有CLAUDE.mdを優先した上で人間に報告する。/context-health-check週次診断に「自分の環境のCLAUDE.local.mdがロールID宣言1行以外を含んでいないかの自己確認」を追加 |

## 2. 設定スニペット（リポジトリ共有 settings.json に入れる最低ライン）

```json
{
  "permissions": {
    "deny": [
      "Read(.env*)",
      "Read(**/*.pem)",
      "Read(**/*.key)",
      "Read(**/secrets/**)",
      "Read(**/credentials*)",
      "Bash(curl * | sh*)",
      "Bash(*--dangerously-skip-permissions*)"
    ]
  }
}
```

- 案件の秘密情報の置き場所に合わせて deny を追加する（/project-init で確定）
- hooks（PreToolUse）でシークレットパターンの送信前検査を追加できる環境なら追加する（/update-config 相当の設定作業として人間が承認）
- Enterprise の場合はこの内容を managed settings 側に移し、個人が外せない状態にする

**破壊防止deny（規模非依存 — 同じく最低ライン）**: 上記が「漏洩」対策なのに対し、こちらは「破壊」対策。破壊事故（force push・履歴消失・テーブル削除・本番リソース削除）は案件規模と無関係に起きるため、1人・ホビー案件でも省略しない:

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf /*)", "Bash(rm -rf ~*)",
      "Bash(git push --force*)", "Bash(git push -f*)",
      "Bash(git reset --hard origin*)",
      "Bash(*DROP TABLE*)", "Bash(*DROP DATABASE*)", "Bash(*TRUNCATE*)",
      "Bash(terraform apply*)", "Bash(terraform destroy*)",
      "Bash(kubectl delete*)", "Bash(aws * delete*)"
    ]
  }
}
```

- 本番系CLI（terraform / kubectl / クラウドCLI）は deny を既定とし、必要なものだけ環境を限定して allow する
- 大規模案件はこれに加えて enterprise-controls §A の第2層（hooks検査）・第3層（破壊的操作の2人ルール・事前バックアップ証跡・ブランチ保護）を適用する

## 3. プラン別の適用チェックリスト（/project-init フェーズ4で実施）

### Enterprise
- [ ] managed settings に deny・MCP許可リストを設定（管理者作業として依頼を起票）
- [ ] 監査ログの確認担当と頻度を決める（ai-dev-standardizer の週次監査に統合）
- [ ] データ保持設定・SSO 前提のアカウント棚卸しを確認

### Team
- [ ] 共有 settings.json をリポジトリにコミット（変更はPRレビュー必須と CLAUDE.md に明記）
- [ ] settings.local.json による上書きの定期検査を /context-health-check に追加
- [ ] メンバーの離任時手順（アカウント無効化・アクセス棚卸し）を運用設計に追加

### 個人（Pro/Max）
- [ ] **業務利用可否の人間承認を記録**（顧客NDA・社内規程との突合結果つき。承認なしで開始しない）
- [ ] 監査ログがない前提の代替統制: 週次で作業ログ（C-01実績記録）と成果物のみから人間がサンプリング監査
- [ ] 離任・契約終了時の扱い（履歴が個人アカウントに残るリスク）を顧客に開示するか人間が判断
- [ ] 機密度の高いデータはそもそも扱わせない設計（マスキング済みデータのみ渡す運用）

## 4. 継続統制（作って終わりにしない）

- security-compliance の定期レビューに「AI利用環境」観点を追加（本標準の1〜11の実地確認）
- /context-health-check 週次: settings の deny が生きているか・.mcp.json に無承認の変更がないか
- 違反検知時: 事故になっていなくても記録（ヒヤリハット台帳）→ ai-dev-standardizer が対策の仕組み化を検討
- 四半期: 本標準自体の棚卸し（プランの機能追加・新しい穴あけパターンの反映）
