# 仮定・未確定管理標準（曖昧さを持ち越すためのルール）

スタートが曖昧な案件（モックのみ・類似サービス参照・ビジネス目的のみ等）では、未解決の曖昧さを抱えたまま
工程を進めざるを得ない。この標準は「曖昧さを消す」のではなく、**曖昧さを構造化して持ち運び、
関係するエージェントが着手時に必ず気づける状態を維持する**ためのルール。

原則: **エージェントに覚えさせるのではなく、作業対象から曖昧さを引けるようにする。**
（コンテキストは揮発するが、機能IDとマーカーは揮発しない）

## 1. 未確定事項の台帳（docs/10-management/open-questions.md の拡張書式）

| 項目 | 内容 |
|---|---|
| Q-ID | Q-001 形式。**全成果物からこのIDで参照される** |
| 質問 | 顧客にそのまま投げられる文面（推奨案つき） |
| 影響範囲 | 機能ID・成果物のリスト（**必須。「全体」は禁止 — 特定できないなら特定する作業を先に起票**） |
| 期限 / 状態 | 起票日・回答期限・状態（未着手/顧客確認中/仮定で進行中/解決/クローズ） |
| 仮定 | 仮定で進める場合のみ: 何を仮決めしたか・根拠・**外れた場合に壊れるもの（爆風半径）** |
| 解決記録 | 回答内容・回答者・日付・巻き戻し作業の起票先 |

## 2. 仮定マーカー（インバンド規約）

仮定の上に作った成果物には、その場所に直接マーカーを埋める:

- 設計書・テスト仕様書: `【仮定: Q-031】経理ロールは参照のみと仮定`
- コード: `// 仮定: Q-031 経理ロールは参照のみ（外れたら権限マトリックス再実装）`
- ER図・テーブル定義: 備考欄に `仮定: Q-031`

ルール:
- マーカーは**仮定に依存する箇所ごと**に置く（文書冒頭に1個ではなく、依存する項目・分岐・カラムに置く）
- マーカーなしで仮定の上に作ることを禁止する（レビュー観点: 報告に「仮定あり」フラグがあるのにマーカーがなければ差し戻し）
- これにより解決時に `grep "Q-031"` で巻き戻し対象が全件列挙できる

## 3. ライフサイクルと担当

```
起票 ──→ 仮定設定 ──→ 伝播 ──→ 解決 ──→ 巻き戻し ──→ クローズ
```

| 段階 | 内容 | 担当 |
|---|---|---|
| 起票 | 曖昧さ検出時に台帳へ（影響範囲必須） | 検出した全エージェント（起票漏れは品質違反） |
| 仮定設定 | 進行可否の判断＋仮定の言語化。爆風半径が大きいものは人間承認 | leader（判断）＋起票者（言語化） |
| 伝播 | 仮定の上で作業する際のマーカー埋め込み | 全生産系エージェント |
| 解決 | 顧客回答の記録。**仮定と回答の一致/不一致を明記** | documentation-specialist |
| 巻き戻し | 不一致の場合: マーカーgrepで影響成果物を全件列挙→修正タスク化（issues起票） | documentation-specialist（列挙）→各担当（修正） |
| クローズ | 巻き戻し完了確認後にクローズ（列挙だけでクローズしない） | leader |

## 4. エージェント共通ルール（起動時フィルタ）

**全生産系エージェント（coder・設計・テスト）は着手前に必ず**:
1. 担当する機能IDで `docs/10-management/open-questions.md` の影響範囲をフィルタする
2. 命中した未解決Qがあれば: 「仮定」が設定済みか確認する
   - 仮定あり → 仮定に従って作業し、マーカーを埋め、報告に「仮定あり: Q-xxx」フラグ
   - 仮定なし → **作業を止めて報告**（勝手に自分で仮定を作らない。仮定設定はleader判断事項）
3. 作業中に新しい曖昧さを発見したら、その場で起票してから続行する

## 5. 曖昧さの経年管理（腐敗防止）

- 「仮定で進行中」が**2週間**更新なし → documentation-specialist が催促起票（D-05と同じ扱い。仮定の放置は納期リスク）
- 仮定の上にさらに仮定が積まれた場合（Q-031の仮定に依存するQ-045）→ 台帳に依存関係を記録し、leader が「仮定の塔」の高さを監視。**2段まで。3段目が必要になったら人間エスカレーション**（そのまま作ると巻き戻し不能になる）
- /context-health-check で毎週: マーカーと台帳の突合（台帳にないマーカー＝孤児 / 解決済みなのに残るマーカー＝巻き戻し漏れ）
- コンテキストヒストリー LATEST.md に「生きている仮定」欄を維持（新セッションが最初に知るべき情報）

## 6. 工程ゲートとの接続

- leader の工程完了判定に追加: 「仮定で進行中」のQを棚卸しし、**次工程に持ち越す仮定を明示的に承認する**（黙って持ち越さない）。爆風半径が次工程で拡大するもの（例: 設計の仮定が実装全体に波及）は持ち越し前に解決を優先する
- 納品前ゲート（/delivery-package）: 未クローズの仮定マーカーが納品物に残っていないか検査。残す場合は「既知の制限事項」として顧客に明示する
