# 大規模SIer統制標準（破壊防止・監査・職務分掌・多段組織・ループ統制）

大規模SIer（多チーム・多段請負・監査対応が前提の組織）で本パックを使うための上位統制。
中小規模案件では ai-security-baseline のみで足りる。適用判定は /project-init（規模・体制の回答）で行う。

思想: **プロンプトで「するな」と書くのは統制ではない。破壊的操作は機械的に止まる（ハーネス）、
止まらなかったものは記録に残る（監査証跡）、記録は別の人間が見る（職務分掌）の三段ではじめて統制。**

## A. 破壊防止ガードレール（3層防御）

### 第1層: 権限deny（settings.json）

破壊防止denyセットの正本は **ai-security-baseline §2「破壊防止deny（規模非依存）」**（破壊事故は規模と無関係に起きるため、全規模共通の最低ラインに置く。中小規模案件もあのセットを省略しない）。大規模ではその上にさらに:

- Enterprise プランでは managed settings 側に置き、個人が外せない状態にする
- 本番系CLIの deny を案件のクラウド・ミドルウェア構成に合わせて網羅し、必要なものだけ環境を限定して allow する（許可の白リスト化を徹底）

### 第2層: hooks（PreToolUse による検査。denyで表現しきれないもの）
- 破壊的SQLパターン（WHERE句なしのUPDATE/DELETE）の実行前ブロック
- 本番を示す環境変数・接続文字列（PROD/本番ホスト名）を含むコマンドの実行前ブロック
- シークレットパターン（鍵・トークン形式）の送信前検査
- hooks の導入・変更自体を承認対象にする（ガードレールを外す変更はPRレビュー＋人間承認）

### 第3層: 運用ルール（機械で止められないもの）
- **破壊的操作の2人ルール**: 本番データ変更・スキーマ変更・一括削除は「起案（AI可）→レビュー（人間）→実行（別の人間）」
- **事前バックアップの証跡**: データパッチ・マイグレーション実行前にバックアップ取得記録がなければ実行しない（incident-responder / data-model-specialist の既存ルールを組織標準に格上げ）
- ブランチ保護（force push禁止・PR必須・レビュー必須）をリポジトリ設定で強制（プロンプトでなくGit側で）

## B. ループ統制（自動で回り続けるエージェントの統制）

ループ（定期実行・自動反復・無人運転）は生産性の最終段だが、**事故も自動で量産する**。開始条件を満たすまでループ化を許可しない:

- [ ] **ハーネス完備が前提条件**: A層のガードレール（deny・hooks・ブランチ保護）が適用済みであること。ハーネスなしのループは禁止
- [ ] **停止手段（kill switch）**: 誰が・どうやって止めるかが文書化され、テスト済み
- [ ] **上限**: 1ループあたりの実行回数・時間・コスト上限を設定（暴走の被害を有限にする）
- [ ] **成果物の扱い**: ループが生成した成果物も同一のQCDゲートを通る（無人生成を理由に検証を省かない。Q-11の二重化はループでも適用）
- [ ] **人間チェックポイント**: 無人で越えてよい境界を定義（例: ループはPR作成まで。マージ・デプロイ・顧客送信は人間）
- [ ] **無人時間帯の制約**: 夜間・休日の無人ループは読み取り系・生成系のみ。書き込み系（コミット・チケット更新）は営業時間内
- [ ] 逸脱検知: ループの実行ログを日次で人間がサンプリング確認（全件目視は不要。異常パターンの検出が目的）

## C. 監査証跡

- **記録の3点セット**: ①AI関与の記録（成果物ごとにAI生成/AI補助/人間作成 — 既存の納品物一覧レベル追跡を全成果物に拡張）②実行記録（C-01の実績記録を監査可能な粒度で: 誰の指示で・どのエージェントが・何を変更）③判定記録（QCDレポート・ゲート判定・リスク受容の承認履歴）
- 保存期間: 案件の契約保存義務に合わせる（最低でも検収後1年を推奨。顧客要件が優先）
- Enterprise プランでは組織の監査ログ機能を一次証跡とし、上記はプロジェクト側の突合材料とする
- 監査で答えられるべき質問を定義しておく: 「この納品コードは誰が書き、誰がレビューし、どのテストで検証されたか」に成果物IDから辿って答えられること（トレーサビリティの背骨がそのまま監査証跡になる）
- **`CLAUDE.local.md`（個人ローカル・gitignore対象）は監査証跡の対象外**になる。体制・承認・リスク受容など意思決定に影響する内容は必ず共有CLAUDE.md・docsへ記録し、CLAUDE.local.mdには置かない（個人の好み・メモに限定。docs/90-pack/standards/ai-security-baseline.md #11）

## D. 職務分掌（SoD）

| 分離 | ルール |
|---|---|
| AI同士 | 生成と検証は別エージェント（Q-11。既存） |
| 人間とAI | AIは起案まで、承認は人間（既存の各エスカレーション） |
| **人間同士（本標準で追加）** | 本番適用・納品確定・リスク受容は「起案した人間」と「承認する人間」を分ける。1人PJでは顧客またはPMOが承認者を務める（自分で書いて自分で承認して自分で本番に入れる、を構造的に不可能にする） |

## E. 多段組織統制（協力会社・BP）

- **アカウント統制**: 協力会社メンバーの個人アカウント持ち込みを禁止。元請が管理するアカウント（Team/Enterprise）を貸与し、離任時に即時無効化
- **同一標準の配布**: パック・QCD基準・規約はチーム（会社）横断で同一版を親リポジトリから配布（tailoring-guide 大規模の項の運用を統制として格上げ）。会社ごとの勝手なテーラリングは禁止し、変更要望は ai-dev-standardizer 経由で一元管理
- **再委託とAI利用の契約整合**: 協力会社にAI利用させる場合、顧客との契約（再委託条項・データ取り扱い）と矛盾しないかを法務確認事項として起票（AIは判断しない）
- **境界の明確化**: 協力会社に開示するデータの範囲・接続できるMCP・触れるリポジトリを会社単位で定義（フィーチャーチームの越境禁止ルールを組織境界にも適用）

## F. 教育・段階的権限

- パック運用者の初期教育3点: ①ai-security-baseline の穴あけ11パターン ②仮定管理（Q-IDとマーカー）③エスカレーション基準（何は人間に聞くか）
- 段階的権限: 新規参画者は最初の2週間「読み取り系＋レビュー受け」のみでパック運用に慣れ、その後書き込み系へ（いきなり全権限を渡さない）
- 教育記録を残す（監査対応。「教育していない人が事故を起こした」は組織の過失になる）

## 適用チェック（/project-init 大規模判定時）

- [ ] A: 大規模向けdenyセット＋hooks＋ブランチ保護の適用
- [ ] B: ループ利用予定の有無を確認し、予定ありなら開始条件チェックリストを運用に組み込む
- [ ] C: 監査証跡の保存期間を契約と突合して確定
- [ ] D: 本番適用・納品の承認者（起案者と別の人間）を体制表で指名
- [ ] E: 協力会社がいる場合、アカウント・データ境界・契約整合の3点を確定
- [ ] F: 参画者教育の実施記録の器を作成
