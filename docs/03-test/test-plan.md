## テストレベル定義（V字対応）

| レベル | 検証根拠 | 実施手段 | 環境 |
|---|---|---|---|
| 単体 | 詳細設計（=実装コード。P2のため） | `test/` のlib関数ユニットテスト | vitest-pool-workers（D1マイグレーション自動適用） |
| **結合** | **基本設計書（docs/02-design/basic-design.md）** | `test/` のAPIハンドラ統合テスト＋E2Eスモーク（`scripts/e2e-smoke.mjs`＝wrangler pages dev実起動でHTTP経由の購入導線検証） | 同上＋ローカルdevサーバー |
| 総合 | 要件定義AC | モック決済フローの手動確認（AGENTS.md）＋E2Eスモーク | ローカル |
| 受入支援 | — | 適用外（顧客・検収なし。実案件組み込み時に実施） | — |

- 開始条件: 基本設計書・ERDが正本として存在すること（本日整備済み）／`npm run typecheck` グリーン
- 完了条件: **Critical/Major未解決 0件**（停止基準: Mustブロッカー0件で次工程可）。不合格は `docs/10-management/issues.md` にI-ID起票
- データ方針: 生成データ・サンプルデータのみ使用（個人情報を含む本番データ使用禁止）。検証後のテストデータは掃除する（AGENTS.md 動作確認フロー5）
- 体制: AI実施=自動テスト実行・結果整理／人間実施=ブラウザでのUI目視確認（任意）

## 不具合管理ルール

- 起票先: `docs/10-management/issues.md`（I-ID）
- 重要度: Critical（データ破壊・セキュリティ・売上直結）/ Major（機能不全）/ Minor / Nit
- 再テスト: 修正後に該当テスト＋`npm test` 全件を再実行し、成績書に再実施行を追記
