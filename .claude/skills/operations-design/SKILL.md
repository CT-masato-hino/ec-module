# 運用設計スキル

運用文書一式を `docs/operations/` に作成する。実作業は operations-designer に委譲する。

## 成果物の構成

```
docs/operations/
├── operations-design.md     ← 運用設計書（体制・定常/非定常一覧・SLA）
├── monitoring.md            ← 監視設計書（項目・閾値・重要度・対応手順参照）
├── procedures/              ← 手順書（1手順1ファイル: 起動停止・リラン・リストア・ユーザー管理…）
├── backup-restore.md        ← バックアップ/リストア設計と試験記録
├── eol-management.md        ← EOL管理表（OS/MW/証明書/ライセンスの期限一覧）
└── handover.md              ← 運用引き継ぎ資料（引き継ぎ会記録・受領確認）
```

## 手順

1. 運用体制（顧客運用/自社受託/無人）とスキルレベルを確認する（手順書の粒度が決まる）
2. operations-designer に運用設計書→監視設計→手順書の順で作らせる
3. 非機能要件（運用・保守性項目）との対応表を必ず含めさせる
4. batch-specialist のリラン手順、incident-responder の障害対応フローを手順書体系に統合する
5. リハーサル: リストア手順・主要非定常手順は実環境で検証し、検証記録を残す
6. 運用部門レビュー → 指摘反映 → 引き継ぎ会 → 受領確認の記録

## 手順書テンプレート（procedures/ 配下共通）

```markdown
# <手順名>                    版数: / 最終検証日: / 検証環境:
- 目的 / 実施契機: / 所要時間: / 実施権限:
- 開始条件（この状態でなければ実施しない）
- 禁止事項（この手順でやってはいけないこと）
## 手順（番号付き。1ステップ1操作。判断分岐は数値基準で）
## 終了確認（観測可能な確認項目）
## 失敗時（切り戻し操作と連絡先）
```

## 原則

- 対応手順のないアラートを設計しない（監視設計と手順書は1対1参照）
- リストア試験なしのバックアップ設計を完成としない
- 手順書は「最終検証日」を持つ（未検証・検証切れは仮版扱い）
- 引き継ぎは受領確認の記録で完了とする（送付やレビュー会開催では完了しない）

## 完了条件

- 非機能要件（運用・保守性）の全項目が運用設計のどこかに対応している（対応表で証明）
- 全アラートに対応手順参照がある
- 主要手順（リストア含む）の検証記録がある
- 運用部門の受領確認がある（納品時）
