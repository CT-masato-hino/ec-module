# 移行計画スキル

移行文書一式を `docs/migration/` に作成する。実作業は migration-specialist に委譲する。

## 成果物の構成

```
docs/migration/
├── migration-plan.md        ← 移行計画書（方式・体制・スケジュール・判定基準）
├── data-mapping.md          ← データ移行マッピング表（現行→新の全項目対応）
├── rehearsal/               ← リハーサル計画・結果（実測時間・問題管理）
├── cutover-runbook.md       ← 当日進行表（分単位タイムチャート・体制・連絡網）
└── rollback-plan.md         ← 切り戻し計画（発動条件・手順・所要時間）
```

## 手順

1. 移行の全体像（現行システム・切替方式・期限・業務停止許容時間）を確認する
2. migration-specialist に移行計画書ドラフトを作らせ、**切り戻し設計を含むこと**を完了条件にする
3. data-mapping は data-model-specialist と共同で作成させる（新モデル側の整合性責任）
4. リハーサル計画（最低2回）をスケジュールに組み込み、実測時間の反映ループを回す
5. 移行判定（GO/NO-GO）資料を作成し、判定は人間＋顧客が行う
6. documentation-specialist に顧客向け説明資料（業務停止時間・顧客側作業）を整形させる

## 移行計画書の必須章立て

1. 移行方針（方式と選定理由・業務影響）
2. 移行対象（データ・ファイル・環境・連携先切替の棚卸し表）
3. 移行体制・役割分担（顧客側作業を明示）
4. スケジュール（リハーサル日程含む）
5. データ移行設計（マッピング・クレンジング方針・検証方式：件数/合計値/サンプル詳細）
6. 当日タイムチャート（分単位・チェックポイント・判断者）
7. 移行判定基準（GO/NO-GO項目・判定時刻・判定者）
8. 切り戻し計画（発動条件・手順・所要時間・切り戻し後の業務）
9. リスクと対策

## 原則

- 切り戻し計画のない移行計画を完成と認めない
- リハーサル未実施の手順を本番に投入しない（実測時間なしのタイムチャートは仮版と明記）
- データクレンジングの責任分担（顧客/ベンダ）を早期に合意し文書化する
- 移行当日の実行は人間。AIは判断材料と手順書の提供まで

## 完了条件

- 全成果物が存在し、リハーサル結果が計画に反映済み
- 判定基準・切り戻し発動条件が数値/事実ベースで定義済み
- 顧客側作業が顧客と合意済み（合意記録あり）
