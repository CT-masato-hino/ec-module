# 詳細設計スキル

詳細設計書を `docs/detail-design/` 配下に作成・更新する。**単体テストの検証根拠になる文書**なので、テスト可能な粒度で書く。

## 前提条件

- 対象機能の基本設計が顧客承認済み（または承認予定日が確定）である
- テーブル定義が data-model-specialist によって確定している

## 手順

1. 対象機能（機能ID）の基本設計書・テーブル定義・既存コードの構造を確認する
2. `docs/detail-design/<機能ID>/` に機能単位で作成する
3. 新規パターン（新しいレイヤー・共通部品・ライブラリ）が必要になったら architecture-guardian の承認を先に取る
4. 完成後、code-reviewer に「基本設計との整合」観点でレビューさせる

## 詳細設計書テンプレート

```markdown
# F-XXX <機能名> 詳細設計書       版数: / 変更履歴:

対応基本設計: <画面ID/IF-ID> / 対応AC: AC-XXX-*

## モジュール構成
（コンポーネント/クラス/ファイルの一覧と責務。既存レイヤリング規約に従う）

## 処理フロー
（主要処理ごと: 入力 → 処理ステップ → 出力。分岐・ループ・例外分岐を明記。
 Mermaidのフローチャート/シーケンス図可）

## メソッド/関数設計
| 名称 | 引数(型) | 戻り値(型) | 処理概要 | 例外/エラー |

## DBアクセス設計
| 処理 | テーブル | CRUD | 条件 | トランザクション境界 | 排他制御 |

## エラーハンドリング設計
（エラー分類 → 捕捉場所 → ログ出力内容 → ユーザー/呼び出し元への返却）

## 単体テスト観点
（この設計から導かれるテスト観点一覧。境界値・異常系を必ず含む。
 /test-planning の単体テスト仕様書の入力になる）
```

## 粒度の指針

- 「実装者が迷わない」ではなく「実装者と別人がテストを書ける」を基準にする
- 既存コードで自明な慣例（例: 標準的なCRUDのRepository実装）は書き写さず「規約に従う」と参照で済ませる。特殊な処理・業務ロジック・分岐にページを割く
- 過剰詳細（コード全文の書き写し）は保守時に二重管理になるため禁止

## 完了条件

- 全処理フローに異常系分岐が定義されている
- DBアクセスにトランザクション境界と排他制御が明記されている
- 単体テスト観点が導出されている（W字対応: 詳細設計の完了条件。/test-planning の単体テスト仕様書の入力になる）
- スコープ除外宣言（「ハンドラ不要」「画面なし」等）と後段の操作記述が矛盾していない（除外宣言をした対象への操作・参照が同一文書内に残っていたら設計内部矛盾として差し戻し）
- 設計書に明記したファイルパス・ハンドラ・ルートの一覧が、実装工程完了時のトレーサビリティ突合（leaderゲート）で使える形になっている
