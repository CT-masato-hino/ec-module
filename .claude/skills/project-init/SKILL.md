# プロジェクト初期化スキル（スターター）

パックを「そのまま全部入り」で使い始めるのを防ぎ、案件に合った最小構成に**根拠つきで**チューニングする。
このスキルはメインセッションが対話で進める（サブエージェントはヒアリングの往復ができない）。記録作成は ai-dev-standardizer に委譲する。

設計の要: **「案件種別（契約の形）」と「出発資産（スタート地点に何があるか）」は別の軸**。
初動工程・ヒアリングの深さ・特有の罠は出発資産で決まる。

## フェーズ0: 配置検査（ヒアリングの前に機械的に確認）

- `docs/90-pack/standards/`（qcd-standards / assumption-management / ai-security-baseline / feature-team）と
  `docs/90-pack/deliverables-catalog.md` `docs/90-pack/tailoring-guide.md` `docs/90-pack/ipa-process-mapping.md` が案件リポジトリに存在するか確認する
- **欠けていればパック原本からコピーしてから進む**（エージェントはこれらを照合先として参照するため、欠けたまま進むと全基準参照が切れる）。コピー元は、クローン導入ならパック原本リポジトリ、プラグイン導入ならプラグインのインストール先（本スキルのファイルパスから遡って特定できる。docs/・CLAUDE.md.template・.mcp.json.template はプラグインルート直下にある）
- `CLAUDE.md.template` / `.mcp.json.template` の有無も確認する

## フェーズ1: ヒアリング（ブロックごとに確認。一度に全部聞かない）

### ブロックA: 案件の骨格
1. 案件種別: 新規開発 / リプレース / エンハンス（既存への機能追加・改修） / 保守運用
2. 契約形態: 請負 / 準委任 / ラボ（工程ごとに異なる場合は内訳。**上流が曖昧な案件で請負ならリスクフラグ**）
3. 受託範囲: どの工程からどの工程まで
4. 規模感（**「概算人月は？」を依頼者に直接答えさせない** — 人月を答えられる依頼者はほとんどいない。答えられる代理指標で聞き、規模帯はこちらが推定して提示する）:
   - 代理指標で聞く: 想定画面数（一覧/入力/帳票の別がわかれば尚可）/ 帳票数 / 外部連携先数 / 利用者数と同時利用の規模 / 扱うデータの件数規模 / 対象業務の種類数 / チーム人数・期間の希望
   - → 回答から規模帯（小: 〜3人月 / 中: 3〜20人月 / 大: 20人月〜）を**こちらが推定して根拠つきで提示**し、依頼者の体感（「もっと軽いはず/重いはず」）とすり合わせる。**ずれたまま次に進まない** — 最初の見立てを誤ると「思ったより時間がかかる/軽すぎる」が全工程の期待値に波及する。確定できなければ仮置きとしてQ-ID起票し、各工程末の見立て再確認（leaderの工程完了判定）で補正する
4-2. 業務影響度（**SLAの入口。「SLAは？」「稼働率は？」と聞かない** — 答えられる依頼者はほとんどいない。業務の言葉で聞き、こちらで変換する。既存番号を壊さないため枝番）:
   - 「このシステムが止まると、誰の・どの業務が・どれくらい困りますか？」
   - 「何時間（何日）止まったら実害が出ますか？夜間・休日も使いますか？」
   - 「データが直前の状態に戻せなくなったら、手作業で復元できますか？」
   - → 回答を影響度レベル（止まっても翌日再開でよい / 当日中に復旧が必要 / 数時間で業務が止まる / 人命・金銭・法令に直結）として記録し、フェーズ3の構成判断（quality-performance・operations-design・incident-responder の採用、Q基準の厳格度）と、要件定義の非機能ヒアリング（/requirements-definition の平易言語質問集）への入力にする

### ブロックB: 出発資産の判定（このスキルの心臓部。複数選択可）
5. スタート地点に何があるか:
   - [ ] 現行システム（稼働中。リプレース/参照対象）
   - [ ] 既存コードベース（エンハンス対象。自社開発か他社開発か）
   - [ ] モック・プロトタイプ・デザインカンプ
   - [ ] 類似システム・参考サービス（「〇〇みたいなものを」）
   - [ ] RFP・要求仕様書・要件リスト
   - [ ] ビジネス目的・課題感のみ（文書なし）
6. 各資産の状態: ドキュメントの有無と鮮度 / コードは読めるか / モックはどこまで作り込まれているか / 参考サービスは触れるか / **資産に本番データ・個人情報・秘密情報（.env・鍵・接続情報）が含まれるか**（含まれる場合はマスキング・除外の方針を決めてからAIに読ませる。フェーズ4ステップ8のdeny設定にも反映。P1現行システム・P2既存コードの初動＝棚卸し・資産評価はAIが資産を読む作業なので、この確認より先に始めない）

### ブロックC: 技術・機能特性
7. 技術スタック（決定済みか、これから選定か）
8. 帳票 / 9. バッチ・締め処理 / 10. 外部連携 / 11. データ移行 の有無

### ブロックD: 顧客・体制
12. 顧客標準（成果物様式・Excel納品指定）と技術リテラシー・意思決定速度
13. 業種規制: 金融(FISC) / 医療(3省2ガイドライン) / 官公庁 / なし
14. 運用の引き取り手 / 15. AI利用の顧客方針（開示要否・データ投入制限。不明なら起票）
16. タスク管理・連絡ツール: Jira / Backlog / Redmine / Notion / Slack の利用有無と、AI同期の可否（→ 使う場合は /pm-sync でセットアップ。外部SaaSへの案件情報送信可否は15と併せて確認）
17. **Claudeのプラン: Enterprise / Team / 個人（Pro/Max）**（→ docs/90-pack/standards/ai-security-baseline.md のプラン別統制を適用する。**個人プランの場合は業務利用可否の人間承認が下りるまで顧客データを扱う作業を開始しない**。機密度の高い業種（Q13が金融/医療/官公庁）×個人プランは原則不可としてエスカレーション）
18. **体制ギャップ（利用者・チームのロール）**: このセッションを実際に操作する人は何人いるか、それぞれの主務ロール（インフラ / PMO・PM / フロントエンド / バックエンド / テスター・QA / ビジネス企画・業務分析 / デザイナー / その他）を確認する。**「みんなが同じ1人のPMとして全エージェントを均等に使う」わけではない**——3人体制（全体管理／業務・フロント・UIUX／非機能・インフラ、等）ならその3分割をそのまま聞く。さらに、フロント／バックエンド／インフラ／QA／PMO／ビジネス企画／デザインの各ドメインについて「専任の人間担当者がいるか（誰か1人が兼ねる場合を含む）」を確認する。→ フェーズ3決定表の「体制ギャップ」行に反映し、専任者不在のドメインほど対応エージェントの意思決定の代行範囲を広げる（UIドメインでの具体例: ui-ux-designerの採用可否）。複数人体制の場合の運用ルール（横断判断は`leader`サブエージェントへ・人間専管事項の窓口・ロールID宣言）は、フェーズ3決定表の該当行とフェーズ4ステップ9で適用する（正本: docs/90-pack/tailoring-guide.md 0-1節）

## フェーズ2: 出発資産別プレイブックの適用

ブロックBの回答に応じて以下を適用する。**資産が複数ある場合は合成する**（開始工程は最も上流の資産に合わせ、他の資産は「要件定義への入力」として登録する）。

### P1: 現行システムあり（リプレース）
- 追加質問: 現行の不満点（=リプレースの動機）/ 現行ドキュメントの信頼度 / 並行稼働の許容
- 初動: **要件定義より先に** legacy-modernizer で機能棚卸し（廃止候補の特定が最大の価値）
- 必須採用: legacy-modernizer / migration-specialist / business-process-analyst
- 罠と対策: 「現行通り」を要件と認めない（必ずAC化）。バグまで踏襲しない（実装挙動と意図仕様を区別）

### P2: 既存コードベースあり（エンハンス）
- 追加質問: 設計書とコードの乖離度（どちらが正か）/ 触ってはいけない領域 / リグレッションテスト資産の有無 / 元の開発会社との関係
- 初動: 資産評価（コード規約・データモデル・テスト資産の現状把握）→ `/context-history` で既存前提を外部化 → 影響分析してから要件定義
- 構成調整: **本パックの規約よりも既存コードの規約を正とする**（coder系への明示指示をCLAUDE.mdに書く）。data-model-specialistは既存ERDの復元から開始
- 罠と対策: 設計書が古い場合はlegacy-modernizerを部分投入（改修対象領域だけ仕様復元）。リグレッション範囲の基準を最初に決める（Q基準に追記）

### P3: モック・プロトタイプだけある
- 追加質問: モックの作者と位置づけ（合意済みデザインか叩き台か）/ モックにない画面・帳票の存在 / 実装技術とモックの乖離（モックのUIライブラリで本実装するのか）
- 初動: requirements-analyst がモックから機能一覧とACを**逆生成**し、曖昧さ検出を全開で実施
- 罠と対策: **モックは「画面の見た目の合意」であって要件ではない**。裏側が全部空白であることを顧客と共有する: 業務ルール・異常系・権限・非機能・バッチ・帳票・データモデル。空白リストを最初の顧客確認事項にする
- 構成調整: frontend-coder はモック踏襲を正とするが「モックにない状態（エラー/空/ローディング）」の設計起票を義務化

### P4: 類似システム・参考サービスがある（「〇〇みたいなの」）
- 追加質問: 参考サービスのどの部分を指しているか（全部ではないはず）/ 自業務と参考サービスの業務の違い / 参考サービスに触れる環境があるか
- 初動: business-process-analyst + requirements-analyst で参考サービスの機能棚卸し → Must/Want仕分け → **自業務との差分こそが本当の要件**として差分定義
- 罠と対策: 「あれと同じで」は要件ではない（同じにできない部分＝データ・規模・業務ルールで必ず崩れる）。デザイン・文言の模倣は著作権・利用規約の観点で「要法務確認」起票。競合の裏側仕様は推測になるため【推定】マーカー管理
- 見積り注意: 参考サービスの完成度を基準に期待値が膨らみがち → estimation-specialist の前提条件に「参考サービスとの差異一覧」を必須添付

### P5: RFP・要求仕様がある（標準ケース）
- 初動: requirements-analyst でRFPの曖昧さ検出（4分類）→ `/requirements-definition`
- 罠と対策: RFPの要求は「書いた人の解釈」を経ている。出典（誰の要求か）を遡って記録する

### P6: ビジネス目的だけある（超上流）
- 追加質問: 目的の定量化は可能か（KPI）/ 予算枠と投資判断者 / 「作らない」も選択肢か
- 初動: **要件定義に入らない**。business-process-analyst で業務・課題の構造化 → システム化する/しない/運用で解決の線引き → estimation-specialist で概算（±50%）→ 投資判断材料の提出まで
- 構成調整: 共通フレームの企画プロセス相当。工程スキルは `/requirements-definition` 以降を保留。契約は準委任を推奨（請負ならスコープ確定リスクをleader経由で人間へ）
- 罠と対策: いきなり画面の話を始めない（顧客はモックを見ると要件が決まった気になる）。PoC/プロトタイプ提案時は「捨てる前提か育てる前提か」を最初に合意

## フェーズ3: 構成の決定（決定表＋プレイブック差分）

共通決定表（規模・帳票・バッチ・連携・規制による採用/保留/除外は従来通り）に、フェーズ2のプレイブック指定を上書き適用し、一覧でユーザー承認を得る:

| 条件 | 構成judgment |
|---|---|
| 常時採用（削減禁止） | leader / documentation-specialist / data-model-specialist(DBあり) / code-reviewer / コンテキスト管理スキル / QCD基準 |
| 規模〜3人月 | requirements-analyst・test-engineer をメイン兼務、estimation/business-process をleader兼務に統合 |
| 帳票なし / バッチなし / 外部連携なし | report-specialist / batch-specialist / api-designer をそれぞれ除外・保留 |
| インフラが顧客側/定型 | infra-coder 除外 |
| 保守受託なし | incident-responder 保留（総合テスト開始時に復帰） |
| 性能要件が緩い | quality-performance 保留（Q-04/Q-08の担保先を code-reviewer に付け替え） |
| Excel納品指定 | /excel-deliverables 採用 |
| UIを持つ案件（Web/モバイル等） | basic-design の**HTMLモックゲート**を適用（画面設計はモックの人間承認を基本設計完了条件にする。バッチ/API専業では適用しない） |
| **体制ギャップ**: 依頼者・チームに専任者がいないドメイン（ブロックD-18。インフラ／PMO／フロント／バックエンド／QA／ビジネス企画／デザインのいずれか） | 該当エージェント（infra-coder／leader／frontend-coder／backend-coder／test-engineer・quality-performance／business-process-analyst／ui-ux-designer）の原則・チェックリスト遵守を厳格運用し、意思決定の代行範囲をCLAUDE.mdに明記する。人間承認は整合性確認に留める |
| **体制ギャップ**: 依頼者・チームに専任者がいるドメイン | 該当エージェントは叩き台生成・効率化ツールとして運用し、専門知識を持つ人間が内容の妥当性まで深く監修する（レビューの厚みは人間側に寄せる） |
| **体制ギャップ**: 操作する人が複数人（ブロックD-18で2人以上と回答） | 各担当者はロールIDつきでCLAUDE.mdのチーム体制表（正本）に登録し、自ドメインの委譲は自分のセッションで完結させる。ドメインを跨ぐ判断は誰でも`leader`サブエージェントに相談すればよく、特定の人間を「Leader」に指名する必要はない。契約・費用・リスク受容・顧客提出・検収判定などAIが判断できない事項だけ「人間専管事項の窓口」を1人指定してエスカレーション先にする。各自のCLAUDE.local.mdには自分のロールIDの宣言1行だけを書く（役割の中身は書かない） |
| （体制ギャップの適用例）UIあり＋専任デザイナー不在（自社・顧客側とも不在） | ui-ux-designer 保留（`.claude/agents.disabled/`へ。復活条件: デザイナー着任）。frontend-coder + requirements-analystがui-ux-designerの原則チェックリストを直接適用する運用に切り替え、担い手をCLAUDE.mdに明記 |
| （体制ギャップの適用例）UIあり＋専任デザイナーあり、または複数チームでUIトンマナ統一が課題 | ui-ux-designer 採用。調査・複数案・提案とHTMLモック承認前レビューを担わせる（フィーチャーチーム運用時は複製禁止・横断共有） |
| 納期・顧客検収のない案件（内製・ホビー・R&D） | **停止基準をCLAUDE.mdのQCD節に確定値で明記**（デフォルト: Mustブロッカー0件なら次工程に進んでよい。残Should/Nitsは工程完了の条件にしない。機序と検知指標は leader の「品質ループの停止判断」） |
| 業種規制あり | security-compliance に規制チェックリストを追記 |
| 機能ドメイン3つ以上＋並行開発 | **フィーチャーチーム運用**を提案（docs/90-pack/standards/feature-team.md。ドメイン専任coderの複製、横断エージェントは複製禁止） |
| 大規模（20人月〜 / 多チーム / 協力会社あり / 監査対応あり） | **enterprise-controls を適用**（docs/90-pack/standards/enterprise-controls.md。破壊防止3層・監査証跡・職務分掌・協力会社統制・ループ統制の適用チェックを実施） |
| ループ運用（定期実行・自動反復）の予定あり | enterprise-controls §B の開始条件チェックリストを適用（**ハーネス未整備でのループ化は禁止**） |
| タスク管理ツール利用あり | /pm-sync セットアップ＋正本ルールの確定（.mcp.json.template から接続） |
| 2案件目以降の組織展開 | ai-dev-standardizer 必須（初案件はleader兼務可）＋ /effort-compression のベースライン設定 |

- 除外は `.claude/agents.disabled/` への移動＋**復活条件**の記録（削除しない）
- **プラグイン導入（README方式A）の場合、エージェント単位の除外はできない**（プラグインは一括enable/disableのみ）。フェーズ3でエージェントの除外・保留が必要と判定された時点で、クローン導入（方式B）への切替を人間に案内する — テーラリングと監査（PRレビュー・audit_pack.py）が前提の案件正式運用は方式Bが正

### 案件別プロジェクト計画HTML（承認はこの絵で行う）

構成が固まったら、テーラリング結果を反映した**案件別プロジェクト計画**を `docs/10-management/project-plan.html` に生成し（`templates/project-plan-template.html` を雛形に）、ブラウザで人間に提示して承認を得る。テキストの決定表だけで承認を取らない — 見えていないゲートは守られない。

- **内容**: 工程フロー（V字/W字）/ 各工程のゲート位置と判定基準 / 成果物マップ（どの工程で docs/ のどこに何が生まれるか）/ 採用・保留・除外エージェント / 現在地ハイライト
- **正本の扱い**: 正本は `docs/10-management/project-phase.md`（工程状態）とテーラリング記録。HTMLはそこから生成する**ビュー**であり、直接編集しない（二重正本を作らない）
- **更新運用**: 工程が進んだとき・テーラリング変更時に再生成して現在地を更新する。工程越境ルール（CLAUDE.md）と連動し、「今どの工程で、次のゲートは何か」を常に人間が見える状態に保つ
- **見た目のルール**: テンプレートのデザイン（チップ・カード・配色）を維持し、**絵文字をアイコン・装飾として追加しない**（アイコンが必要ならテンプレート同様のモノクロSVGで）
- `tools/docs_viewer.py`（ローカルWebビューア）導入時はそのトップページとして扱う

## フェーズ4: 適用

1. `CLAUDE.md.template` → `CLAUDE.md` 生成（出発資産と適用プレイブック名を明記。埋まらない項目は【要確認】起票）
2. 除外エージェントの移動・QCD基準の全IDテーラリング確定（ai-dev-standardizer主導）・確定値のCLAUDE.md転記
3. docs骨組み生成: `docs/10-management/open-questions.md` / `docs/10-management/issues.md` / `docs/10-management/decisions/` / `docs/10-management/context-history/`（初期LATEST.mdに本セットアップを記録）/ `docs/10-management/project-phase.md`（工程状態の正本。`templates/project-phase.md` から生成し、初期工程・開始日を記入）
4. プレイブック固有の初期成果物を登録（P1:棚卸し表の器 / P3:モック空白リスト / P4:差分定義表 / P6:線引き表）。**この時点で判明している曖昧さは全件、Q-ID＋影響範囲つきで open-questions に起票する**（曖昧なスタートの案件ほど、初期起票の量がそのまま仮定管理の土台になる。正本: docs/90-pack/standards/assumption-management.md）
5. テーラリング記録（採用/保留/除外＋根拠＋復活条件）を CLAUDE.md へ（ai-dev-standardizer）
6. 開発規約の準備: 技術スタック確定済みなら `/dev-standards` を初動リストに入れる（**coder稼働前に規約正本を作る**。未確定なら /tech-stack-selection の完了を規約生成のトリガーとして記録）。テストランナー（vitest/jest等）の設定には `.claude/worktrees/**` の除外をデフォルトで含める（並行セッションのworktree成果物をテストに巻き込まない）
7. 工数圧縮の土台: `/effort-compression` のStep1-3（作業棚卸し・移管度マップ・ベースライン設定）をセットアップ期に実施（後からベースラインは作れない）
8. **AIセキュリティ統制の適用（省略不可）**: `docs/90-pack/standards/ai-security-baseline.md` のプラン別チェックリストを実施 —
   共有 settings.json への deny 設定（漏洩系: .env・鍵・secrets の読み取り禁止 ＋ **破壊系: force push・DROP/TRUNCATE・本番系CLI等の破壊防止deny。破壊事故は規模と無関係に起きるため1人・ホビー案件でも省略しない** — ai-security-baseline §2 の2スニペットを適用）/ MCP許可リストの確定 /
   秘密情報の置き場所ヒアリングと deny への追加（ブロックB質問6の資産機微度の回答を反映）/ 報告免責ルールのCLAUDE.md明記。
   Enterprise なら managed settings への移設を管理者タスクとして起票、個人プランなら業務利用承認の記録を確認。
   **このステップが完了するまで coder 系エージェントを稼働させない**
9. **複数人体制の場合**: 体制表（ロールID×担当者×主務ドメイン×主に使うエージェント）をCLAUDE.mdに記載し、「人間専管事項の窓口」（契約・費用・リスク受容・顧客提出・検収判定の最終エスカレーション先）を1人指定する（**この表が役割定義の正本。PRレビューとaudit_pack.pyの対象**。横断判断そのものは`leader`サブエージェントに任せてよく、担当者を「Leader」に指名する必要はない）。各自には`CLAUDE.local.md`（プロジェクトルート、`.gitignore`対象）の作成を案内し、内容は「私のロールID: X」の1行宣言だけに限定する（**役割の中身・QCD基準・権限・工程ルール・エスカレーション基準はCLAUDE.local.mdに書かない**。書式外の記述は無効扱いとし共有CLAUDE.mdを優先する。docs/90-pack/standards/ai-security-baseline.md #11）。`.gitignore`に`CLAUDE.local.md`が含まれているかを確認する

## フェーズ5: 検収と最初の一手

- [ ] CLAUDE.md の {{ }} が全て埋まっている（【要確認】は起票済み）
- [ ] 出発資産と適用プレイブックが記録されている

- [ ] AIセキュリティ統制（プラン別チェックリスト＋deny設定）が適用済み。個人プランの場合は業務利用の人間承認が記録されている
- [ ] 人間がテーラリング一覧を承認済み（承認は `docs/10-management/project-plan.html` の計画図上で行う）

最初の一手はプレイブックの「初動」をそのまま提示する（P1:現行棚卸し / P2:資産評価 / P3:モック逆生成＋空白リスト / P4:参考サービス棚卸し / P5:RFP曖昧さ検出 / P6:業務構造化。複数資産なら最上流から）。

## 原則

- 回答を推測で埋めない。「とりあえず全部入り」を提案しない
- 出発資産の見立てを間違えたと気づいたら、途中でもプレイブックを切り替えてテーラリング記録に残す
- このスキルの実行自体が最初のADR（`docs/10-management/decisions/YYYYMMDD-project-setup.md`）になる
