# PM連携スキル（タスク管理ツール・Slack同期）

外部ツールと docs/ の二重管理を防ぎながら、マネジメント側（人間・顧客）が普段のツールで進捗を見られる状態を作る。

## セットアップ（初回のみ）

1. `.mcp.json.template` を案件リポジトリ直下に `.mcp.json` としてコピーし、使うツールだけ残す
2. サーバーの最新情報は mcp registry で確認する（`suggest_connectors` / `search_mcp_registry`。テンプレのURL・パッケージ名は陳腐化する前提）
3. 認証（OAuth/APIキー）は**人間が実施**。APIキーは環境変数経由にし、`.mcp.json` への直書き・コミット禁止
4. **接続してよい情報の範囲を先に決める**: 顧客名・案件情報を外部SaaSに送ることになるため、CLAUDE.mdのAI利用方針・顧客秘密保持契約と突合する（security-compliance確認 → 人間承認）。判断がつくまで同期は「ローカル→外部の起票」を止め、読み取りのみで運用する

## 正本ルール（最重要。二重管理禁止原則の適用）

| 情報 | 正本 | docs/ 側の扱い |
|---|---|---|
| タスクの状態（誰が・いつまで・進行状態） | **外部ツール** | `docs/10-management/issues.md` はAI作業用ミラー（セッション開始時に取り込み） |
| 経緯・理由・技術文脈（なぜ） | **docs/**（ADR・設計書） | 外部ツールのチケットからはリンクで参照 |
| 顧客確認事項（open-questions） | docs/ が正本 | 顧客が見るツールに転記する場合は「転記先」を台帳に記録 |
| 進捗の定義・集計 | QCD基準（D-01） | 外部ツールのステータスをD-01（0/100）にマップする |

- ID対応を必ず残す: `ISS-012 ↔ PROJ-345` 形式で issues.md 側に外部キーを持つ（Q-ID仮定マーカーとの連結もこのIDで辿れるようにする）
- 状態がツールとdocsで食い違ったら**外部ツールを正として上書き**し、食い違いの原因（同期漏れ）を記録する

## 同期対象マッピング

| docs/ 側 | 外部ツール側 | 方向・タイミング |
|---|---|---|
| issues.md の起票・状態変更 | 課題チケット | 双方向。外部→ローカル: セッション開始時に取り込み / ローカル→外部: 起票・変更の都度 |
| 議事録の宿題（/meeting-minutes） | チケット起票（担当・期限つき） | ローカル→外部。議事録確定と同時 |
| QCD週次レポート | Slack投稿（サマリ＋詳細リンク） | 週次。leader判定後 |
| 障害（/incident-report） | インシデントチケット＋Slack即時通知 | 発生時。第一報と同期 |
| スクワッド別進捗（feature-team運用時） | ボード/エピックに機能IDを対応付け | 週次 |

## ツール別の設定メモ（初回に確定して CLAUDE.md に記録）

- **Jira**: プロジェクトキー / 課題タイプのマップ（issues→Task、障害→Bug、顧客確認→独自タイプ） / D-01に使うステータス（Done以外は全部0%扱い）
- **Backlog**: スペース・プロジェクトID / 種別・マイルストーンのマップ（マイルストーン=工程）
- **Redmine**: トラッカーのマップ / 進捗率フィールドは使わない（D-01と矛盾するため。ステータスのみ使用）
- **Notion**: 課題DBのプロパティマップ（Status/Assignee/Due/機能ID列を必須にする）
- **Slack**: 通知の粒度ルールを最初に決める — 即時通知はエスカレーション（C-02/D-02の警告・障害）のみ、それ以外は日次/週次サマリに集約（**通知疲れはアラート疲れと同じ失敗**）

## 運用の型

- **セッション開始時**: 外部ツールから自分担当（AI作業分）のオープンタスクを取り込み → docs/10-management/issues.md ミラー更新
- **セッション終了時**: 状態変更・作業記録を外部ツールへ反映（C-01の実績記録と同時に）
- **転記作業そのものをAIが担う**ことで管理系工数を圧縮する（/effort-compression の管理系・移管度Aの代表例）

## 原則

- 外部ツールに書く内容は顧客・他社も読む前提で書く（内部略語・AI痕跡・率直すぎるリスク表現を避け、リスクは事実ベースで）
- 同期できない期間（認証切れ等）は「同期停止中」を明示し、復旧後に差分同期する（黙って古い状態を見せない）
- ツール側の一括変更（ステータス大量更新・削除）は人間承認を必須とする
