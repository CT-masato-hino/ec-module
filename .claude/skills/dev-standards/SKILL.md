# 開発規約スキル

規約の正本を `docs/90-pack/standards/dev-standards/` に生成・維持する。
**このスキルの存在理由**: coder系は「既存コードの規約に従う」が原則だが、新規案件には既存コードがない。規約なしでcoderを稼働させると「1人目のcoderの癖」が事実上の規約になる。**coder稼働前に規約を確定させること**。

## 生成フロー

1. **入力の確認**: 技術スタック（確定済みが前提）/ 既存コードの有無 / 会社標準規約の有無 / 顧客規約の有無
2. **生成方式の選択**:
   - 既存コードあり（エンハンス）→ コードから規約を**抽出**（リバース。実態と理想を混ぜない — 実態を規約化し、直したい点は「改善予定」として分離）
   - 会社/顧客標準あり → それを正として案件固有の差分だけ作る
   - 何もなし（新規）→ スタックのデファクト標準をベースに、下記カタログの必須項目を対話で確定
3. **レビュー**: architecture-guardian が一貫性・実現性をレビュー → 確定をADR化
4. **登録**: CLAUDE.md に規約正本のパスを記載。coder/reviewer は以後これを照合先にする
5. **機械化**: 規約のうちlint/formatter/CIで強制できる項目は設定ファイルに落とす（**書いた規約より強制される規約**。機械化した項目は規約文書側に「lint強制」と明記し、レビュー観点から外す）

## 規約カタログ（12種。案件に応じて取捨、各規約は Must / 推奨 / 例 の3層で書く）

| # | 規約 | 必須項目（これが埋まるまで完成と言わない） |
|---|---|---|
| 1 | coding-fe.md | 命名 / コンポーネント分割基準 / 状態管理の使い分け / スタイル方式 / 禁止事項（直DOM操作等） |
| 2 | coding-be.md | レイヤリングと依存方向 / 命名 / DI方針 / Tx境界の置き場所 / 禁止事項 |
| 3 | api-conventions.md | URL設計 / HTTPステータスと業務エラーコードの2層体系 / ページング / versioning（api-designerと共同） |
| 4 | db-conventions.md | 命名（テーブル/カラム/インデックス） / 型の使い分け / 共通カラム / 論理削除・排他方式（data-model-specialistと共同） |
| 5 | logging.md | レベル定義 / 必須項目（誰が・何を・結果） / 禁止出力（個人情報・秘密情報） / 保持期間 |
| 6 | error-handling.md | エラー分類（業務/システム） / エラーコード採番体系 / ユーザー向けメッセージの原則 / リトライ方針 |
| 7 | git-workflow.md | ブランチモデル / コミットメッセージ規約 / PRテンプレート / マージ条件（Q-01/Q-02と連動） / worktree・ブランチのライフサイクル（マージ後はブランチを離れる・未コミット残骸の扱い・長期未コミットworktreeの掃除ルール） / レビュー指摘のトリアージ（Must=マージ不可。Should/Nitsは**「記録」と「起票」を分離** — その場で直した方が早い軽微なものはissue行を作らずコミットメッセージへの記録で足りる。禁止は「暗黙の握りつぶし」だけで「全件issue化」ではない。leaderの品質ループ停止判断と連動） |

## 作り込みの運用（生成して終わりにしない）

- **規約違反の扱いを最初に決める**: code-reviewer の指摘区分と対応（Must=規約のMust違反）
- **例外手続き**: 規約から外れたい場合は architecture-guardian 承認＋期限付き例外としてADR化（無断逸脱の禁止）
- **改訂**: 「同じ指摘が3回出たら規約の欠陥」として改訂対象に。改訂は変更履歴＋全coderへの周知（CLAUDE.md更新）
- **棚卸し**: 工程完了ごとに「守られていない規約」を集計 → 直すか捨てるか判定（ai-dev-standardizerの形骸化点検と連動）
- **フィーチャーチーム運用時**: 規約はチーム間共有の生命線。ドメイン別の方言を作らせない（docs/90-pack/standards/feature-team.md 参照）

## 完了条件

- 採用した規約すべてで必須項目が埋まり、Must/推奨/例の3層になっている
- 機械化可能な項目がlint/CI設定に反映され、規約文書に「lint強制」マークがある
- architecture-guardian レビュー済み・ADR化済み・CLAUDE.mdに正本パス登録済み
- coder系エージェントの照合先が「既存コード」から「規約正本＋既存コード」に切り替わっている
