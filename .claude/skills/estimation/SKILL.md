# 見積りスキル

見積り根拠書一式を `docs/estimation/` に作成する。実作業は estimation-specialist に委譲する。

## 手順

1. 見積り種別を確認する: 概算（±50%）/ 詳細（±10-20%）/ 変更見積り。契約形態も確認する
2. estimation-specialist に以下を順に作らせる:
   - `docs/estimation/YYYYMMDD-<種別>/wbs.md` — 成果物ベースのWBSと規模カウント
   - `docs/estimation/YYYYMMDD-<種別>/basis.md` — 算定根拠（難易度区分・生産性基準・出典）
   - `docs/estimation/YYYYMMDD-<種別>/assumptions.md` — 前提条件・除外事項・変更管理の取り決め
3. 積上げと類推のクロスチェック結果（乖離と原因）を確認する
4. leader に「工数レンジ＋リスク」でレビューさせ、最終サマリを人間へ渡す（金額化・値引きは人間）

## 見積りサマリテンプレート（人間への引き渡し形式）

```markdown
# 見積りサマリ（YYYY-MM-DD / 種別: 概算|詳細|変更）

## 工数レンジ
| ケース | 工数 | 主な想定 |
|---|---|---|
| 楽観 | | 全前提が守られた場合 |
| 標準 | | リスク係数込みの推奨値 |
| 悲観 | | 主要リスク顕在時 |

## 工程別内訳（標準ケース）
| 工程 | 工数 | 規模根拠（画面N・帳票N・IF N 等） |

## 前提条件トップ5（崩れた場合の影響つき）
## 除外事項
## 最大リスク3件とバッファの置き方
## 人間が判断すべきこと（単価・金額・営業判断）
```

## 原則

- 単一数字で出さない（必ずレンジ＋前提）
- 隠しバッファ禁止（バッファは明示行で管理）
- 変更見積りは「作る工数」と「作り直す工数（手戻り）」を分けて計上する
- 請負案件の見積り提出は必ず人間の承認を経る（勝手に顧客へ出さない）

## 完了条件

- 全数字に出典（規模×生産性基準 or 実績比）がある
- 前提条件・除外事項が列挙され、変更管理の取り決めがある
- クロスチェックの乖離が説明されている
