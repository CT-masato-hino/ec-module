# 障害報告書スキル

顧客向け障害報告書を段階に応じて作成する。事実整理は incident-responder、文書化は documentation-specialist に委譲する。保存先: `docs/incidents/YYYYMMDD-<事象slug>/`

## 3段階の報告

### 第一報（発生直後。速さ優先）
```markdown
# 障害発生のご報告（第一報）  YYYY-MM-DD HH:MM 時点
1. 事象: （何ができない状態か。業務の言葉で）
2. 影響範囲: （対象業務・対象ユーザー・件数。不明なら「調査中」と明記）
3. 発生日時・検知経路:
4. 現在の対応状況:
5. 次報予定: HH:MM
```
- **原因は書かない**（「調査中」とする。推測の原因を書いて後で覆すのが最悪）
- 発生から第一報までの目標時間は CLAUDE.md の顧客報告ルールに従う

### 中間報（暫定対処後）
- 第一報の構成＋「暫定対処と現在の業務可否」「原因の調査状況（確定部分のみ）」「復旧見込み」

### 最終報（恒久対策確定後）
```markdown
# 障害報告書  版数: / 提出日:
1. 事象概要
2. 影響範囲（確定値: 期間・件数・対象・データ影響の有無と回復結果）
3. 経緯（時系列表: 分単位・事実のみ。検知/報告/対処/復旧の各時刻）
4. 原因（直接原因と根本原因を区別。図解推奨）
5. 暫定対処（実施済み内容と実施日時）
6. 恒久対策（内容・適用予定日）
7. 再発防止策（仕組みレベル: テスト追加・チェック自動化・監視追加・レビュー観点追加）
8. 水平展開（同種リスクの調査範囲と結果）
```

## 手順

1. 段階（第一報/中間報/最終報）を確認する
2. incident-responder から事実（時系列・影響数値・原因）を受け取る
3. documentation-specialist が顧客リテラシーに合わせて文書化する
4. **提出前に人間の承認を必須とする**（謝罪文言・補償に関わる表現は人間が決める）
5. 最終報の再発防止策は `docs/10-management/issues.md` にタスク起票し、実施を追跡する

## 原則

- 時系列は事実のみ・分単位・空白なし（「この間何をしていたのか」は必ず聞かれる）
- 再発防止策に「注意する」「ダブルチェックする」だけの項目を入れない（仕組みに落ちないものは不可）
- 影響件数は確定前は「調査中」、確定後は根拠（特定クエリ）つき
- 責任・補償・契約に関わる記述はドラフトに含めず、人間判断の欄として空ける

## 完了条件

- 該当段階のテンプレート全項目が事実で埋まっている（推測の混入なし）
- 人間の承認を得ている
- 最終報の場合: 再発防止策が全件タスク化されている
