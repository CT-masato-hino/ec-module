# コンテキスト自己診断スキル

コンテキスト汚染の兆候を検出し、`docs/10-management/context-history/health-YYYYMMDD.md` に診断結果を残す。調査はサブエージェント（Explore / documentation-specialist）に委譲し、メインセッションには結果サマリのみ返す。

## 診断項目

### 1. 文書間矛盾チェック
- 要件（AC）↔ 基本設計 ↔ 詳細設計 ↔ テスト仕様のトレーサビリティ断絶
- ERD/テーブル定義 ↔ 実マイグレーションの不一致
- ADR（docs/10-management/decisions/）↔ 実装の不一致（決定と違うパターンが使われていないか）
- `docs/10-management/project-phase.md` の宣言工程 ↔ 実作業の乖離（直近コミットのtype・成果物の出現場所が宣言工程と合っているか。乖離は「黙った越境」として越境記録の追記＋工程正本の更新を促す）

### 2. 鮮度チェック
- 各正本文書の最終更新日と、対応するコードの最終変更日の乖離（コードだけ進んで文書が古い箇所）
- open-questions.md に期限切れ・塩漬け（2週間以上更新なし）の課題がないか
- context-history/LATEST.md が1週間以上前でないか

### 3. 外部化漏れチェック（会話内にしかない知識）
- 直近の作業で「決定したのにADR/設計書に反映されていないもの」がないかをユーザーに質問形式で確認する（最大5問。例: 「Xの方式はYで進めていますが、この決定はどこにも記録されていません。ADR化しますか？」）

### 4. 自己矛盾の兆候
- 同一概念に複数の名前が使われていないか（用語のゆれは汚染の初期症状）
- 廃止したはずの方式・却下した案がドキュメントに残存していないか
- 自分の環境に `CLAUDE.local.md` があれば内容を確認し、「私のロールID: X」の1行宣言以外を含んでいないか（QCD基準・権限・工程ルール・エスカレーション基準に関する記述等があれば書式外として共有CLAUDE.mdのチーム体制表が正である旨を報告し、是正を促す。docs/90-pack/standards/ai-security-baseline.md #11）

### 5. 仮定の健全性（docs/90-pack/standards/assumption-management.md との突合）
- `【仮定: Q-xxx】` / `仮定: Q-xxx` マーカーを docs/ とコードから grep し、台帳と突合する:
  - 台帳にないマーカー（孤児仮定）→ 起票漏れとして復元
  - 解決済みQのマーカー残存 → 巻き戻し漏れとして issues 起票
  - 台帳で「仮定で進行中」なのにマーカーゼロ → 伝播漏れ（仮定の上で作ったのに印がない）の疑い
- 仮定の依存段数を確認（2段超は即エスカレーション）

## 出力

```markdown
# コンテキスト健全性診断 YYYY-MM-DD

## 総合判定: 健全 / 注意 / 要リセット
## 検出事項（重要度順・各1行＋修正先ファイル）
## 推奨アクション
（「注意」→ 修正タスクリスト / 「要リセット」→ /context-history で大リセットを推奨）
```

## 判定基準

- **健全**: 矛盾ゼロ、鮮度乖離が軽微
- **注意**: 矛盾1〜3件、または外部化漏れあり → その場で修正
- **要リセット**: 矛盾4件以上、または用語のゆれ・却下案の残存など構造的な汚染 → 月次を待たず大リセット
