# テスト計画スキル

テスト文書一式を `docs/03-test/` 配下に作成する。作業は test-engineer サブエージェントへ委譲する。

## V字対応（必須）とドラフト作成時期（W字・シフトレフト）

| レベル | 検証根拠 | ドラフト作成時期 | 成果物 |
|---|---|---|---|
| 単体 | 詳細設計書 | 詳細設計の完了前（単体テスト観点として詳細設計書に含める） | `docs/03-test/unit/<機能ID>.md` |
| 結合 | 基本設計書 | 基本設計の完了前 | `docs/03-test/integration/<シナリオID>.md` |
| 総合 | 要件定義書(AC) | 要件定義の完了前（テスト計画書全体もこのタイミング） | `docs/03-test/system/<シナリオID>.md` |
| 受入支援 | 要件定義書(AC) | 総合テスト開始前 | `docs/03-test/uat-support/` |

- **テスト設計を実装後に回さない。** ドラフトの目的は、テスト観点から左側工程へフィードバックを返すこと（ACのテスト容易性・完了条件の実測可能性・テストデータの準備可否）。「テスト可能なACを書く」規律の受け手を工程の最後まで空席にしない
- ドラフトは対応する定義・設計工程の完了条件に含まれる（leaderのゲート判定・deliverables-catalog と連動）
- 各テスト工程の開始時に、ドラフトを実施可能な確定版へ更新する（環境・データ・実施者を確定）

## 手順

1. 対象工程・対象機能を確認し、テスト計画書を先に作る（初回のみ。**要件定義の完了前に着手する**）
2. test-engineer にテスト仕様書（左側工程ではドラフト）を作成させる（根拠文書の項番トレース必須）
3. ドラフト作成で検出したテスト困難点（観測不能なThen・実測不能な完了条件・準備不能なデータ）を、対応する定義・設計工程に**完了前に**フィードバックする
4. トレーサビリティ検査: 「根拠なきテスト」「テストなき要件/設計項目」の両方向を検出し、リストを報告させる
5. 実施後、成績書を記録する。不合格は `docs/10-management/issues.md` に起票する

## テスト計画書テンプレート（`docs/03-test/test-plan.md`）

- 対象範囲・対象外範囲（明記必須）
- テストレベルごとの: 開始条件 / 完了条件（消化率・不具合密度・未解決Critical件数） / 環境 / データ方針（個人情報を含む本番データ使用禁止） / 体制（AI実施と人間実施の分担）
- 不具合管理ルール（起票先・重要度定義・再テスト手順）

## テスト仕様書 兼 成績書テンプレート

| No | 根拠項番 | 観点分類 | 前提条件 | 手順 | 期待結果 | 合否 | 実施日 | 実施者 | エビデンス |
|---|---|---|---|---|---|---|---|---|---|

- 観点分類: 正常系 / 境界値 / 異常系 / 権限 / 同時実行 / データバリエーション
- 期待結果は観測可能な事実のみ（「正しく〜される」禁止）
- エビデンス（スクリーンショット・ログ）は `docs/03-test/evidence/` に保存しパスを記録

## 完了条件（工程共通）

- トレーサビリティ両方向に漏れゼロ、または漏れの理由が記録済み
- テスト計画書の完了条件を満たす、または未達理由と対応方針をleader判断で記録済み
- 成績書に実施日・実施者・エビデンスが全件記録されている
