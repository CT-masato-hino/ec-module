# 工数圧縮スキル

「AIを使ったからなんとなく速い」を「**どの作業が・何割・なぜ圧縮されたか**」の計測に変える。
SESモデルでは工数=原価。圧縮の成果は再投資（品質強化・追加提案・要員配置）の判断材料として人間に渡す。

## 大原則

1. **ベースラインなき圧縮率は認めない**（比較対象がなければ圧縮は主張できない）
2. **Q基準の悪化を伴う圧縮は無効計上**（テストを薄くして「圧縮」は粉飾。QCD週次レポートとセットで評価）
3. **圧縮率の使い道はビジネス判断＝人間**（準委任で工数請求している場合、圧縮の顧客への見せ方は契約に直結する。AIは材料まで）

## 手順

### Step 1: 作業棚卸し（案件開始時）
WBSから作業を分類し、`docs/90-pack/standards/effort-map.md` に台帳化する:

| 作業分類 | 例 | 従来工数の目安（ベースライン） |
|---|---|---|
| 文書系 | 設計書作成・議事録・報告資料・課題管理 | 画面設計書1本=X時間 等 |
| 実装系 | コーディング・単体テスト・レビュー | 画面1本=X時間 等 |
| テスト系 | テスト仕様作成・実施・エビデンス整理 | ケース作成N件/時間 等 |
| 調査系 | 現行調査・技術調査・不具合調査 | （最も見積りにくい。実績蓄積で更新） |
| 管理系 | 進捗集計・会議準備・ツール転記 | 週X時間 |

### Step 2: AI移管度マップ
各作業分類に移管度を設定する（docs/ai-standards/raci.md と整合させる）:
- **A: AI自律**（人は結果確認のみ）— 議事録整形・進捗集計・トレーサビリティ検査・Excel変換など
- **B: AI主働＋人レビュー** — 設計書ドラフト・実装・テスト仕様・調査報告
- **C: 人主導＋AI補助** — 顧客折衝資料・アーキテクチャ判断・見積り
- **D: 人のみ** — 契約・検収判定・顧客への約束
- 移管度マップの整備率100%が C-06 の基準。「まだAに上げられるB」を四半期ごとに棚卸しする（圧縮パイプライン）

### Step 3: ベースライン設定
- 優先順: 自社の過去実績 > estimation-specialist の生産性基準 > 業界一般値（出典明記）
- ベースラインの恣意的な水増し禁止（圧縮率を盛るためにベースラインを重くしない。設定根拠を記録し ai-dev-standardizer が監査可能にする）

### Step 4: 月次計測（C-07）
- 実績工数（C-01の記録から）をベースラインと比較し、作業分類別の圧縮率を算出
- 同時にQ基準の状態を併記（**Q黄/赤の期間の圧縮は「要検証」として分離**）
- レポート: `docs/90-pack/standards/effort-report/YYYYMM.md`

```markdown
# 工数圧縮レポート YYYY-MM
## サマリ: 全体圧縮率 X%（有効計上分。要検証分 Y% は除外）
## 作業分類別: 分類 / ベースライン / 実績 / 圧縮率 / Q状態 / 主因
## 圧縮パイプライン: 次に移管度を上げる候補と必要な準備（規約整備・テンプレ化・ツール連携）
## 人間への判断材料: 浮いた工数の再投資選択肢（品質強化 / 前倒し / 追加提案 / 体制縮小）と各リスク
```

### Step 5: 圧縮の構造化（単発で終わらせない）
圧縮が確認された作業は「なぜ圧縮できたか」を分解し、再現可能にする:
- テンプレ化できたから → パック/規約に還元（ai-dev-standardizer経由）
- ツール連携で転記が消えたから → /pm-sync の同期対象に追加
- AIの検証コストが下がったから → Q-11の二重化実績と突合（検証を省いた圧縮でないことを確認）

## 連携

| 関係先 | 内容 |
|---|---|
| qcd-standards C-06/C-07 | 移管度マップ整備率・圧縮率の基準値と判定 |
| estimation-specialist | ベースライン提供・C-05再見積りへの圧縮実績の反映（**次案件の見積り競争力になる**） |
| ai-dev-standardizer | 移管度マップの監査・圧縮ノウハウのパック還元 |
| leader → 人間 | 再投資判断・顧客への圧縮の見せ方（契約影響） |
