# ERD管理スキル

ERDとテーブル定義の正本を `docs/erd/` 配下で維持する。**このスキルを通さないテーブル変更は禁止**（CLAUDE.mdにも明記されている前提）。

## 正本の構成

```
docs/erd/
├── erd.md               ← ERD本体（Mermaid erDiagram形式。ツール非依存で差分レビュー可能）
├── tables/<テーブル名>.md ← テーブル定義書（1テーブル1ファイル）
├── codes.md             ← コード定義書（区分値・列挙値）
└── CHANGELOG.md         ← 変更履歴（日付・変更内容・理由・影響機能・承認者）
```

## 手順（変更時）

1. 変更理由・要求元機能IDを添えて data-model-specialist に委譲する
2. data-model-specialist が影響調査（参照する全機能・帳票・バッチ・外部IF）を行い、変更案を返す
3. 破壊的変更（列削除・型変更・NOT NULL追加等）はユーザーへエスカレーションし、承認を得てから確定する
4. 確定後: erd.md / tables/ / CHANGELOG.md を更新し、マイグレーション方針を backend-coder に引き渡す

## テーブル定義書テンプレート

```markdown
# <テーブル物理名>（<論理名>）      版数: / 変更履歴: CHANGELOG参照

## 概要・所有機能ドメイン
## カラム定義
| No | 物理名 | 論理名 | 型 | 桁 | NULL | デフォルト | PK/FK/UK | 個人情報 | 説明 |
## インデックス（名称・列・種別・根拠となる検索要件）
## 制約・整合性ルール（FK・CHECK・アプリ層で守る整合性）
## 共通カラム準拠（作成日時/更新日時/版数/論理削除 等のプロジェクト規約）
```

## 整合性監査（定期実行: 結合テスト前・納品前に必須）

data-model-specialist に以下を検査させる:
- ERDとテーブル定義書と実DBスキーマ（マイグレーション）の三者一致
- コード定義書と実装内の区分値の一致
- 個人情報フラグ列と security-compliance のデータ保護設計の一致
- 孤立テーブル（どの機能からも参照されない）・未定義参照の検出

結果は監査レポートとして `docs/erd/audit-YYYYMMDD.md` に保存する。
