# 要件定義スキル

要件定義書一式を `docs/01-requirements/` 配下に作成・更新する。作業は原則 requirements-analyst サブエージェントへ委譲し、このスキルはテンプレートと手順を定める。

## 手順

1. 入力を確認する: 顧客要望・RFP・議事録・既存システム資料。不足していればユーザーに何があるか確認する
2. requirements-analyst に以下を順に作らせる:
   - `docs/01-requirements/functional-list.md` — 機能一覧（後述テンプレ）
   - `docs/01-requirements/acceptance-criteria/<機能ID>.md` — 機能ごとのAC
   - `docs/01-requirements/non-functional.md` — 非機能要件（IPA非機能要求グレード6大項目）
   - `docs/10-management/open-questions.md` — 曖昧さ指摘・確認事項（顧客に投げられる質問文で）
3. 曖昧さ指摘のうち重要なものトップ5をユーザーに提示し、顧客確認の要否を仰ぐ
4. **テスト計画のシフトレフト（W字）**: 要件定義の完了前に `/test-planning` を呼び、テスト計画書（全体）と総合テスト仕様書ドラフト（要件ACベース）を作成する。ドラフト作成で見つかった「テストできないAC」「実測できない完了条件」は要件側に差し戻して完了前に直す
5. documentation-specialist に変更履歴・版数を整えさせる

## Acceptance Criteria テンプレート

```markdown
# F-XXX <機能名> 受入基準

## ユーザーストーリー
<役割>として、<目的>のために、<操作>したい

## 受入基準
### AC-XXX-01: <観点名>
- Given（前提）:
- When（操作）:
- Then（期待結果）: ※観測可能な事実で書く。数値・表示文言・状態遷移

### 異常系・境界値
（エラー時挙動・権限なし時・境界値を必ず1件以上定義する。未定義なら open-questions へ）

## 運用・管理者観点
（管理者が使い慣れた操作性か。運用担当者のスキルレベル前提を明記）
```

## 非機能ヒアリング質問集（平易言語版 — SLA用語で顧客に質問しない）

「SLAは？」「稼働率は？」「RTO/RPOは？」に答えられる顧客・依頼者はほとんどいない。**専門用語で聞くのは「わからない」を「未定義」として固定する行為**であり、後から「思っていたのと違う」を生む。業務の言葉で聞き、回答を要求値に**こちらが変換**する（/project-init ブロックA質問4-2の業務影響度の回答があれば出発点にする）:

| 聞きたい要求値 | 顧客に聞く質問（業務の言葉） | 変換先 |
|---|---|---|
| 稼働率・運用時間帯 | 「システムが止まると、誰の・どの業務が・どれくらい困りますか？」「夜間・休日も使いますか？」 | 可用性: 稼働率・計画停止 |
| RTO | 「障害が起きたら、何時間（何日）以内に復旧しないと実害が出ますか？」 | 可用性: 復旧目標 |
| RPO | 「直前何分（何時間）分の入力データなら、消えても手作業で入れ直せますか？」 | 可用性: 復旧時点目標 |
| レスポンスタイム(p95) | 「一番よく使う画面で、何秒待たされたら仕事にならないですか？」 | 性能: p95目標値 |
| 同時接続・ピーク | 「月末・締め日など一番混む時は、何人が同時に使いますか？」 | 性能: 同時接続 |
| データ量成長 | 「1日（1月）にどれくらいのデータが増えますか？何年分残しますか？」 | 性能: データ量・保持 |
| バックアップ | 「データが全部消えたら、事業は続けられますか？紙や他システムから復元できますか？」 | 運用: バックアップ方式 |
| 監視・障害検知 | 「止まっていることに、誰がどうやって気づけばよいですか？（利用者からの電話でよいか）」 | 運用: 監視設計 |
| セキュリティ要求 | 「このデータが漏れたら、誰にどう謝ることになりますか？（個人情報・取引先情報・社外秘の別）」 | セキュリティ: 機密度分類 |

- 回答は上表の変換先に落とし、根拠列に「ヒアリング回答（誰が・いつ）」を記録する（推奨値の仮置きと区別する）
- 「わからない」の場合は推奨値を仮置きして Q-ID 起票する。**専門用語のまま顧客に宿題として投げない**（答えられない宿題は返ってこない）

## 非機能要件テンプレート（IPA非機能要求グレード準拠）

6大項目それぞれについて「要求値・根拠・確認状態」を表で定義する:

| 大項目 | 項目例 | 要求値 | 根拠(顧客合意/推奨値) | 状態 |
|---|---|---|---|---|
| 可用性 | 稼働率・計画停止・復旧目標(RTO/RPO) | | | |
| 性能・拡張性 | レスポンスタイム(p95)・同時接続・データ量成長 | | | |
| 運用・保守性 | 監視・バックアップ・保守時間帯・リリース方式 | | | |
| 移行性 | 移行方式・並行稼働・切り戻し | | | |
| セキュリティ | 認証方式・暗号化・監査ログ・アクセス制御 | | | |
| システム環境・エコロジー | 設置環境・耐障害・省エネ | | | |
| システム環境・エコロジー | **環境面数（開発/検証/ステージング/本番の何面構成か）・面ごとの用途とデータ方針** | | | |
| システム環境・エコロジー | **主要ミドルウェアの構成方針（スキーマ設計・主要パラメータ等）** | | | |

- 顧客と未合意の項目は推奨値を仮置きし、状態を「仮置き（要顧客合意）」とする。仮置きのまま設計工程を終えない
- 環境面数・ミドルウェア構成方針は自由記述に埋めず、明示行として必ず記入する（「小規模なので1面」も判断として記録）
- 技術選定へ委ねる項目は状態を「仮置き（/tech-stack-selection の構成デフォルト逸脱チェックで確定）」と受け渡し先まで明記し、確定したら本表の状態を更新する（委ねたのに拾われず落ちる事故の防止。消し込み責任は tech-stack-selection 側）

## 完了条件

- 全機能にACが存在し、Thenがすべて観測可能な形式である
- 非機能6大項目に「未記入」がない（仮置きは可）
- open-questions の Must 機能に関わる項目がゼロ、または顧客確認中である
- テスト計画書（`docs/03-test/test-plan.md`）と総合テスト仕様書ドラフトが存在し、全Must機能のACがドラフトにトレースされている（W字対応）
