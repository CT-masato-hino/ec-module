# 議事録スキル

会議の生メモ・文字起こしから議事録を作成し、**決定・宿題を管理台帳へ反映するところまで**を1セットで行う。作業は documentation-specialist に委譲する。

## 手順

1. 入力（メモ・文字起こし・録音の要約）と会議情報（日時・出席者・会議体名）を受け取る
2. documentation-specialist に議事録を作成させる: `docs/minutes/YYYYMMDD-<会議体>.md`
3. **台帳反映（このスキルの本体価値）**:
   - 決定事項 → 設計・要件に影響するものはADR化提案 / 要件変更は requirements-analyst の影響分析へ
   - 宿題（AI/自社側） → `docs/10-management/issues.md` へ起票（担当・期限つき）
   - 宿題（顧客側） → `docs/10-management/open-questions.md` へ（期限・催促予定日つき）
   - 発言と決定の矛盾（過去決定との衝突）を検出したら報告する
4. 顧客送付用がある場合は体裁を整え、送付前に人間の確認を必須とする

## 議事録テンプレート

```markdown
# <会議体名> 議事録
日時: YYYY-MM-DD HH:MM–HH:MM / 場所: / 出席者: （敬称・社名つき）/ 記録者:

## 決定事項（この会議で確定したことのみ。提案・雰囲気の合意は書かない）
| No | 決定内容 | 根拠・経緯 | 影響（要件/設計/スケジュール） |

## 宿題事項
| No | 内容 | 担当（社名・氏名） | 期限 | 起票先 |

## 協議内容（争点と両論を簡潔に。発言は帰属つき「○○様:」）

## 次回: 日時 / 主要議題
```

## 原則

- 「決定」と「継続協議」を厳密に区別する（曖昧な合意を決定と書くと後で揉める）
- 発言の帰属を残す（特に顧客の仕様言及。後日の「言った言わない」対策）
- 議事録に書いて終わりにしない: 台帳反映まで完了して初めて完了（反映件数を報告する）
- 顧客に不利な表現・内部事情（体制の火の車等）を顧客送付版に残さない

## 完了条件

- 決定・宿題が全件台帳へ反映され、反映先が議事録に記載されている
- 過去決定との矛盾チェックが実施済み
