# コンテキストヒストリースキル

セッションを超えた知識の継続を担保する資料 `docs/10-management/context-history/` を作成・更新する。作業は documentation-specialist に委譲する。

## 成果物

```
docs/10-management/context-history/
├── LATEST.md               ← 常に最新の完全版（新セッションはこれだけ読む）
└── archive/YYYYMMDD.md     ← 更新時点のスナップショット（時系列差分チェック用）
```

## LATEST.md テンプレート

```markdown
# コンテキストヒストリー（更新日: YYYY-MM-DD）

## 現在地
- 工程: docs/10-management/project-phase.md 参照（正本。ここに転記しない — 転記は陳腐化する） / 今週のゴール: / 直近の完了物:

## 主要決定事項（新しい順・最大20件。古いものはADRへの参照に置き換える）
| 日付 | 決定 | 理由の要旨 | ADR |

## 制約・前提（変わらないもの）
- 顧客制約 / 技術制約 / 契約上の制約

## 進捗サマリ
| 機能ID | 要件 | 基本設計 | 詳細設計 | 実装 | 単体 | 結合 | 状態メモ |

## オープン課題（docs/10-management/open-questions.md の要約＋期限つきトップ5）

## 生きている仮定（「仮定で進行中」のQ全件: Q-ID・仮定内容・爆風半径・マーカー箇所数）
（新セッションが最初に知るべき情報。ここに載っていない仮定の上で作業してはいけない）

## 直近の落とし穴・注意点
（次のセッションがハマりそうなこと。例: 「X画面の仕様は口頭変更済みで設計書が古い」）

## 参照すべき正本
（ERD・設計書・課題管理表へのパス一覧）
```

## 発火頻度の基準

- 基準は**週次更新＋月1の大リセット**。加えて工程ゲート通過時は必ず更新する
- 高密度な開発（1日に複数セッション）では `LATEST.md` の随時更新は構わない。ただし `archive/` のスナップショットは**1日1本まで**（同日2回目以降は同日ファイルを上書きする）。アーカイブは時系列差分チェックのための基準点であり、細かく刻むほど差分チェックのノイズが増えて本来の矛盾検出が埋もれる
- アーカイブの生成が週3本を超えるペースで続く場合、それは引き継ぎ需要の増加ではなく品質ループの自己増殖（起票→再検証→再起票）のサインであることが多い。leader の「品質ループの停止判断」チェックリストで一度立ち止まる

## 更新手順

1. 更新前に現在の `LATEST.md` を `archive/YYYYMMDD.md` にコピーする
2. documentation-specialist に、今回のセッション/期間の決定・進捗・課題を反映させる
3. **時系列差分チェック**: 前回スナップショットとの差分を確認し、次を検出する:
   - 過去の決定と矛盾する新決定（→ 意図的な変更ならADR化、無自覚ならコンテキスト汚染の兆候として報告）
   - 消えた課題（解決記録なしに消えたものは復活させる）
4. 差分チェック結果を1行サマリでユーザーに報告する

## 月1大リセットの手順

1. このスキルで LATEST.md を完全更新する
2. `/context-health-check` を実行し、矛盾を解消してから確定する
3. 新セッションの最初の指示を「`docs/10-management/context-history/LATEST.md` を読んで再開」とする
